## Context

We are building an Emdros corpus for Vendergood, a reconstructed constructed language
(W. J. Sidis, ~1905). The current corpus is small: ~10 worked sentences from addition
entries, each demonstrating specific grammatical features of the language. Backend is
SQLite3. Scale is modest — this will likely stay under a few thousand Word objects for
the foreseeable future.

## The specific decision

**Modeling compound words.**

Vendergood has productive compounding: two roots fuse to form a new word
(`temporpont` = tempor + pont, "bridge across time"; `alterfin` = alter + fin,
"completion of another's work"). About a third of current entries are compounds.

**Option A — flat Word features:**

```
[Word lemma="temporpont" root_1="pont" root_2="tempor" ...]
```

Single `Word` object. Compound structure is carried as features on the word.
Query for "words containing root pont": `[Word root_1="pont"] OR [Word root_2="pont"]`.
Clean for two-root compounds; requires `root_3` if triple compounds ever arise
(none currently exist in Vendergood, but the morphology permits them).

**Option B — nested Compound sub-object:**

```
[Word lemma="temporpont"
  [Compound root="pont" position=1]
  [Compound root="tempor" position=2]
]
```

Accurate nesting; MQL can query roots positionally; scales to triple compounds
without schema change. Adds object type complexity.

## The tradeoff as I see it

For this corpus at this scale, Option A is almost certainly sufficient. The query
"give me all words whose primary root is X" is the main use case, and flat features
handle that cleanly. Option B would be worth the complexity if we had phrase-level
compound analysis (e.g., coreferential chains across compounds) — but Vendergood
syntax is agglutinative enough that that kind of analysis isn't likely in the
near term.

**Specific question:** Have you seen flat root features cause problems in a comparable
small corpus — one where the primary query pattern is "find words sharing a root"
rather than deep syntactic annotation? Or is the flat representation fine for our scale,
with the understanding that a migration to nested objects would be the upgrade path
if the corpus grows?
